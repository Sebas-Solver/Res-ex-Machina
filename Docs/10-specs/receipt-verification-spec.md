## 1. Receipt Hash (receipt_hash)

The `receipt_hash` is a SHA-256 fingerprint that deterministically links the key fields of a record. It is the value anchored to the blockchain.

### Algorithm

```
receipt_hash_algo: "sha256"
receipt_canonicalization: "pipe-separated"
```

### Calculation

```
input = record_id | content_hash | agent_wallet_lowercase | nonce | created_at_iso8601
receipt_hash = "sha256:" + HEX(SHA-256(input))
```

### Canonicalization Rules (NORMATIVE)

| # | Rule | Detail |
|---|---|---|
| 1 | **Separator** | Pipe character `|` (U+007C), without preceding or trailing spaces |
| 2 | **agent_wallet** | Converted to **lowercase** BEFORE concatenation. Example: `0x1234AbCd...` → `0x1234abcd...`. The `0x` prefix is kept |
| 3 | **created_at** | Exact **ISO 8601 UTC** format: `YYYY-MM-DDTHH:MM:SS.mmmZ`. Always with 3 millisecond digits. Always in UTC (`Z` suffix). Example: `2026-02-14T14:32:54.661Z`. Variations with offset (`+00:00`) or without milliseconds are not accepted. **Note:** `created_at` is generated by the RxM server upon record acceptance (see section 5: Trust Model) |
| 4 | **Encoding** | The entire string is **UTF-8** encoded before applying SHA-256 |
| 5 | **No padding** | No spaces, newlines, or extra characters between fields or at the start/end |
| 6 | **Prefix** | The result is prefixed with `sha256:` followed by 64 lowercase hexadecimal characters |

### Example

```
Input:  "019c5c91-a485-74f5-be0b-4941c49ac507|sha256:aed327...06ac|0x1234567890abcdef1234567890abcdef12345678|v4n1rx9v3gassh4k59135f1fgkdp6l3i|2026-02-14T14:32:54.661Z"
Output: "sha256:ba56c69ce7076b79809467637fb70b3c6a7f35bf7038204bc3f325586d56d8d4"
```

### Pseudocode (for implementations in any language)

```python
# Python
import hashlib

canonical = "|".join([
    record_id,                          # UUID v7
    content_hash,                       # "sha256:{64hex}"
    agent_wallet.lower(),               # "0x{40hex}" lowercase
    nonce,                              # string
    created_at,                         # "2026-02-14T14:32:54.661Z"
])

digest = hashlib.sha256(canonical.encode("utf-8")).hexdigest()
receipt_hash = f"sha256:{digest}"
```

```go
// Go
canonical := strings.Join([]string{
    recordID,
    contentHash,
    strings.ToLower(agentWallet),
    nonce,
    createdAt,
}, "|")

hash := sha256.Sum256([]byte(canonical))
receiptHash := fmt.Sprintf("sha256:%x", hash)
```

## 2. EIP-712 Signature (PoG Bundle)

The PoG bundle signature uses the [EIP-712](https://eips.ethereum.org/EIPS/eip-712) standard (typed structured data signing).

### Domain

```json
{
    "name": "ResExMachina",
    "version": "1",
    "chainId": 0,
    "verifyingContract": "0x0000000000000000000000000000000000000000"
}
```

> **IMPORTANT:** `chainId: 0` and `verifyingContract: 0x0000...0000` are the **real production values**, NOT placeholders. These values are used because the RxM EIP-712 signature is **off-chain**: it is not tied to any deployed contract or specific chain. The signature certifies the agent's identity, not a contract interaction.

### Types

```json
{
    "PoGBundle": [
        { "name": "schema", "type": "string" },
        { "name": "content_hash", "type": "string" },
        { "name": "agent_wallet", "type": "address" },
        { "name": "model_id", "type": "string" },
        { "name": "runtime_id", "type": "string" },
        { "name": "process_type", "type": "string" },
        { "name": "human_intervention_level", "type": "uint8" },
        { "name": "pipeline_steps", "type": "uint16" },
        { "name": "timestamp", "type": "string" },
        { "name": "nonce", "type": "string" }
    ]
}
```

### Field Mapping

The PoG bundle has `generation_process` as a nested object, but the EIP-712 signature requires flat fields. The mapping is:

| Field in PoG bundle | Field in EIP-712 message |
|---|---|
| `generation_process.process_type` | `process_type` |
| `generation_process.human_intervention_level` | `human_intervention_level` |
| `generation_process.pipeline_steps` | `pipeline_steps` |
| All other fields | Copied directly |
| `signature` | **Excluded** from the message (it is the output) |
| `generation_process` (object) | **Excluded** (flat fields are used) |

### Verification

To verify the signature:
1. Reconstruct the flat `message` from the `pog_bundle` fields (see mapping above)
2. Use `verifyTypedData` (viem), `eth_signTypedData_v4` (ethers), or equivalent
3. Check that the recovered address matches `agent_wallet`

## 3. Blockchain Anchoring

### Method

```
anchor_method: "calldata"
```

The `receipt_hash` is stored on the blockchain as a UTF-8 string in the `data` (calldata) field of an ordinary transaction.

### Technical encoding details

```
1. receipt_hash string (e.g. "sha256:20b13a4c...")
2. → encode as UTF-8 bytes
3. → convert each byte to its hexadecimal representation
4. → prefix with "0x" to form the calldata
5. → send as tx.data in a 0 ETH value transaction
```

**Concrete example:**

```
receipt_hash:  "sha256:20b13a4c3459ce4c..."
UTF-8 bytes:   [0x73, 0x68, 0x61, 0x32, 0x35, 0x36, 0x3a, 0x32, 0x30, 0x62, ...]
                 s      h      a      2      5      6      :      2      0      b
tx.data (hex): "0x7368613235363a32306231336134633334353963653463..."
```

The calldata is the **complete string** `"sha256:{64hex}"` encoded as UTF-8 and then to hexadecimal. It is **not** the raw 32-byte binary digest.

### On-chain Verification

```python
# Pseudocode
tx = get_transaction(anchor.tx_hash)
calldata_hex = tx.input[2:]                     # remove "0x"
calldata_bytes = bytes.fromhex(calldata_hex)     # hex → bytes
calldata_text = calldata_bytes.decode("utf-8")   # bytes → string

assert calldata_text == anchor.anchored_hash     # "sha256:20b13a..."
```

### Anchoring Transaction

| Field | Value |
|---|---|
| `to` | `FEE_RECEIVER_ADDRESS` (RxM address) |
| `value` | `0` (transfers no ETH) |
| `data` | `receipt_hash` encoded as UTF-8→hex |
| `from` | `ANCHOR_WALLET` (RxM service wallet, not the agent's) |

## 4. Declaration Fields (unverified by the platform)

The following fields are **signed agent declarations**, unverified by RxM:

| Field | Description | Verification |
|---|---|---|
| `model_id` | Identifier of the AI model used | ❌ Signed declaration |
| `runtime_id` | Execution environment identifier | ❌ Signed declaration |
| `process_type` | Type of generative process | ❌ Signed declaration |
| `human_intervention_level` | Level of human intervention (0-4) | ❌ Signed declaration |
| `pipeline_steps` | Number of pipeline steps | ❌ Signed declaration |

These fields are cryptographically signed (EIP-712) by the `agent_wallet`, ensuring that **the agent declared those values**, but not that they are necessarily truthful. Verifying the veracity of `model_id` is planned for future versions.

## 5. Trust Model

RxM is a **technical attestation protocol**. It is crucial to understand what it guarantees and what it doesn't.

### What RxM Guarantees (technical proof)

| Guarantee | Mechanism |
|---|---|
| **Signer identity** | The `agent_wallet` that signed the PoG bundle is cryptographically verified via EIP-712. It is impossible to forge the signature without the private key |
| **Content integrity** | The `content_hash` (SHA-256) links the record to specific content. Any content modification produces a different hash |
| **Time of acceptance** | `created_at` is generated by the RxM server the instant it accepts the record. This value is part of the `receipt_hash` and is therefore linked to the blockchain anchoring |
| **Existence at a given time** | Blockchain anchoring establishes an **upper time bound**: the record existed at the latest in the block where it was anchored |
| **Immutability** | Once created, a record cannot be modified or deleted (INV-001, INV-002) |
| **No-replay** | The `nonce` is unique per wallet (database constraint), preventing signature reuse |

### What RxM Does NOT Guarantee (agent declaration)

| Field | Why it's not verified |
|---|---|
| **`model_id`** | RxM cannot verify which AI model actually executed. The agent declares it and RxM records it |
| **`runtime_id`** | RxM has no access to the agent's execution environment |
| **`human_intervention_level`** | It's a self-declaration; RxM cannot measure human intervention |
| **`process_type`** | Self-declaration of the generative process type |
| **`pog_bundle.timestamp`** | The agent declares its own timestamp. RxM records `created_at` as the acceptance time, but does not verify the agent's timestamp |

> **Legal Analogy:** An RxM receipt is comparable to a **notarial affidavit**: the notary (RxM) certifies that a person (agent_wallet) declared something (pog_bundle) at a given time (created_at), and seals it immutably (anchor). The notary does NOT certify that what was declared is true.

### Temporal semantics: three distinct moments

```
┌─────────────────────┐    ┌──────────────────────┐    ┌────────────────────────┐
│ pog_bundle.timestamp │ →  │ created_at           │ →  │ anchor.anchored_at     │
│ Declared by agent    │    │ Set by server        │    │ Set by blockchain      │
│ ❌ Unverified        │    │ ✅ Part of receipt    │    │ ✅ Immutable on-chain   │
│                      │    │    hash              │    │                        │
└─────────────────────┘    └──────────────────────┘    └────────────────────────┘
```

`created_at` is generated by the RxM server upon record acceptance and is part of the `receipt_hash` material. This implies the server participates in the deterministic receipt process. The blockchain establishes an independent upper temporal bound verifiable by anyone.

```json
        "amount": "0.00020000",

        "tx_hash": "0xabcdef...9876",
        "chain_id": 84532,
        "to": "0xABCDabcdABCDabcdABCDabcdABCDabcdABCDabcd"
    },
    "anchor": {
        "tx_hash": "0xcb5529f3...",
        "block": 37655645,
        "chain_id": 84532,

```

> **Note on `eip712_domain`:** `chain_id: 0` and `verifying_contract: 0x0000...0000` are the **real production values**, NOT placeholders. The RxM EIP-712 signature is off-chain by design: it certifies the agent's identity without binding to any deployed contract. This allows the same signature to be valid regardless of whichever network it is anchored on.

> **Note on `spec_version`:** Indicates the version of this specification the receipt was generated with. If in the future canonicalization rules, anchor_method, or EIP-712 types change, the verifier must apply the rules corresponding to the receipt's `spec_version`.

## 7. Official Test Vector

This vector allows any implementation to verify that its code produces the correct results.

### Inputs

```
record_id:     "test-0001-0001-0001-000000000001"
content_hash:  "sha256:e3b0c44298fc1c149afbf4c8996fb92427ae41e4649b934ca495991b7852b855"
agent_wallet:  "0x1234567890abcDEF1234567890abcDEF12345678"
nonce:         "test-nonce-alpha-001"
created_at:    "2026-01-01T00:00:00.000Z"
```

### Expected receipt_hash

```
Canonical input (pipe-separated):
"test-0001-0001-0001-000000000001|sha256:e3b0c44298fc1c149afbf4c8996fb92427ae41e4649b934ca495991b7852b855|0x1234567890abcdef1234567890abcdef12345678|test-nonce-alpha-001|2026-01-01T00:00:00.000Z"

Expected output:
"sha256:e4490f45f9c87381073210b696046c30d6caeaea844b25e8d8294c46c63abc6c"
```

> The reference hash: `e3b0c44298fc1c149afbf4c8996fb924...` is the SHA-256 of the empty string (""), a standard known value.

### How to use this vector

```bash
# Verify with Node.js
echo -n 'test-0001-0001-0001-000000000001|sha256:e3b0c44298fc1c149afbf4c8996fb92427ae41e4649b934ca495991b7852b855|0x1234567890abcdef1234567890abcdef12345678|test-nonce-alpha-001|2026-01-01T00:00:00.000Z' | sha256sum

# If your implementation produces the same hash, it is correct.
```
