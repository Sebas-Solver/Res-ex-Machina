# DOCUMENTO DONDE AÑADO IDEAS, FRASES, COSAS QUE SE ME OCURREN PARA EL PROYECTO
De momento son ideas que se van asentándo. Están sujetas a cambios. 


No solo es un registro o un storage de contenido. 

Respecto de otras plataformas, la diferenciación real no está en:
- blockchain
- timestamps
- almacenamiento
- detección
- NFTs
- compliance

Está en esto (en una sola idea):
Res ex Machina es el primer registro neutral y automatizado donde los agentes de IA dejan trazas verificables de lo que generan, bajo su propia identidad técnica.

Ninguna de las alternativas que listaste hace estas cosas a la vez:
- Registro positivo (no detección)
- Identidad del agente IA (no del humano ni de la plataforma)
- Metadata completa del proceso de generación
- Neutralidad jurídica (registro de hechos, no de derechos)
- Está automatizado, es decir, el registro no depende de validación humana ex ante.
- La validación humana es excepcional, no constitutiva.

Ese es el núcleo.

# Artefactos para el agente de Antigravity
Para que el agente de Antigravity ejecute bien, el siguiente artefacto debería ser uno de estos:
1. PRD v1 “agent-proof”
– schemas, endpoints, invariantes, out-of-scope
2. Spec técnica PoG v1 completa (EIP-712 + ejemplos)
3. Lista de invariantes del sistema (“esto nunca puede pasar”)

## PRD v1 "agent-proof"
(Product Requirements Document, pero pensado para un agente)
Qué es: Es el documento principal de ejecución.
Define exactamente qué entra en el MVP v1 y qué NO entra.
Para un humano, un PRD puede ser ambiguo.
Para un agente, la ambigüedad es peligrosa.

Qué contiene (estructura ideal)
A) Objetivo del MVP (1 párrafo)
Muy literal:
“El objetivo del MVP v1 es permitir a agentes de IA registrar eventos de generación de outputs, dejando una traza verificable (hash + PoG + timestamp) consultable públicamente, sin validación humana ex ante.”

Esto alinea todo.

B) Scope IN (UIDADO (lo que SÍ se construye)

Ejemplo:
```Yaml
scope_in:
  - register_generation_event
  - agent_identity_via_wallet
  - pog_v1_bundle_signed
  - immutable_timestamp_anchor
  - public_verification_by_hash
  - fee_base_required
  - json_receipt_export
```
Esto es oro para un agente.

C) Scope OUT (crítico)
Esto evita que el agente “sea creativo”.
```Yaml
scope_out:
  - ai_detection
  - originality_scoring
  - human_validation_flows
  - content_curation
  - social_features
  - semantic_analysis
  - copyright_or_ip_claims
```
Sin esto, el agente puede añadir cosas “porque parecen útiles”.

D) Endpoints exactos

Nada de “una API para registrar”.

```Yaml
endpoints:
  - POST /v1/records
  - GET /v1/records/{id}
  - GET /v1/records/verify?hash=
  - GET /v1/records/{id}/export
```

E) Schemas versionados

Ejemplo:

```Yaml
schemas:
  - Record.v1
  - PoGBundle.v1
  - Agent.v1
```

F) UX operativa mínima

Muy importante para humanos:

```Yaml
ui:
  purpose: operational_review
  views:
    - records_table
    - record_detail
    - verify_by_hash
  forbidden_actions:
    - approve
    - reject
    - edit_metadata
```

## Spec técnica PoG v1 completa
(Proof of Generation como estándar técnico)
Qué es: 
Es la definición formal de cómo una IA demuestra que generó algo.

No es “una feature”.
Es el corazón probatorio del sistema.

Por qué necesita documento propio

Porque si no lo separas:
- el agente puede mezclarlo con lógica de negocio
- o cambiarlo “para mejorar”

PoG debe ser estable, aburrido y verificable.

Qué contiene
A) Definición formal

Ejemplo:

Proof of Generation v1 is a signed, canonical bundle that attests that a specific output hash was generated by a specific AI agent at a specific time under declared conditions.

B) Schema exacto (canonical JSON)
{
  "schema": "pog.v1",
  "content_hash": "0x...",
  "agent_wallet": "0x...",
  "model_id": "provider:model:version",
  "runtime_id": "runtime_hash",
  "generation_process": {
    "process_type": "direct|pipeline|iterative",
    "human_intervention_level": 0
  },
  "timestamp": "ISO-8601"
}

C) Firma (EIP-712)
Aquí le dices al agente exactamente:
- qué campos se firman
- cómo se hashéan
- cómo se verifica

Incluyes:
- ejemplo de payload
- ejemplo de firma válida
- ejemplo de verificación fallida

D) Qué NO hace PoG

Muy importante:
```Yaml
pog_does_not:
  - prove_originality
  - detect_ai
  - assert_legal_authorship
```

## 3. Lista de invariantes del sistema
(“Esto nunca puede pasar”)
Qué es: Es el cinturón de seguridad lógico del sistema.

No define features.
Define límites absolutos.

Un agente respeta invariantes mucho mejor que “buenas intenciones”.

```Yaml
invariants:
  - A record, once created, can never be deleted
  - A record hash can never be modified
  - No human action can mark a record as "valid"
  - No record can exist without a fee being paid
  - A PoG bundle must always be signed by the agent identity
  - The platform never assigns authorship or ownership
```
Dónde se usan
- tests
- validaciones
- revisiones automáticas
- reasoning del propio agente
